1. CloudTrail — CRITICAL

1.1 No CloudTrail Trails Found

Field Detail
Severity 🔴 CRITICAL
Service CloudTrail
Resource N/A

Description: No AWS CloudTrail trails exist in this environment. CloudTrail is the primary mechanism for recording API calls and management events across AWS services. Without it, there is no audit log of user activity, resource changes, privilege escalations, data access events, or any other actions taken within the account.

Impact: - Complete lack of forensic capability — if a breach has occurred or is occurring, it cannot be investigated. - Compliance violations across most major frameworks (SOC 2, ISO 27001, PCI-DSS, HIPAA, CIS AWS Foundations). - Inability to detect or attribute malicious or accidental changes to the environment. - All other HIGH severity findings in this report have been operating in an unmonitored environment.

Recommendation: - Immediately enable a multi-region CloudTrail trail covering all regions. - Enable log file integrity validation. - Deliver logs to a dedicated, access-controlled S3 bucket with versioning and MFA delete enabled. - Forward logs to CloudWatch Logs or a SIEM for alerting on suspicious activity. - Ensure the trail covers both management events and, where appropriate, data events (S3, Lambda).

2. IAM Findings — HIGH

2.1 Overly Permissive Policy: ai_scanner_user

Field Detail
Severity 🟠 HIGH
Service IAM
Resource ai_scanner_user (IAM User)

Description: The IAM user ai_scanner_user has been assigned a managed policy that grants excessive permissions beyond what is required for its function.

Impact: - Compromise of this user (e.g., via credential theft or key exposure) could allow an attacker to perform far-reaching actions across the AWS environment. - Violates the principle of least privilege.

Recommendation: - Audit the permissions currently granted to ai_scanner_user. - Replace the overly permissive policy with a custom, scoped policy granting only the specific actions and resources required. - Evaluate whether this should be an IAM user at all; consider using an IAM Role with short-lived credentials instead. - Rotate or audit any existing access keys associated with this user.

2.2 Overly Permissive Policy: terraform-bootstrap

Field Detail
Severity 🟠 HIGH
Service IAM
Resource terraform-bootstrap (IAM User/Role)

Description: The terraform-bootstrap principal has an overly permissive managed policy attached. Terraform bootstrap identities often carry broad permissions to provision infrastructure and are a high-value target.

Impact: - If credentials are exposed (e.g., in source code repositories, CI/CD pipelines, or environment variables), an attacker could gain near-administrative access to the AWS environment. - Blast radius of a compromise would span all services the policy permits.

Recommendation: - Restrict permissions to only those services and actions required during the bootstrap phase. - Consider breaking permissions into separate roles for different Terraform operations (e.g., plan vs. apply). - Ensure credentials are not stored in plaintext in version-controlled code. - If this is an IAM user, strongly consider migrating to an IAM Role assumed by a CI/CD pipeline with short-lived credentials (e.g., via OIDC federation).

2.3 Overly Permissive Policy: Developers Group

Field Detail
Severity 🟠 HIGH
Service IAM
Resource Developers (IAM Group)
Affected Users hughie, sasha, david, daphne, john

Description: The IAM group Developers has an overly permissive managed policy, affecting five users: hughie, sasha, david, daphne, and john. All members of this group inherit these excessive permissions.

Impact: - Any one of the five developer accounts being compromised provides an attacker with excessive access to AWS resources. - Developers may be able to access sensitive data, modify security configurations, or escalate their own privileges. - The risk is multiplied by the number of users — five separate accounts represent five potential attack vectors.

Recommendation: - Review and scope down the managed policy attached to the Developers group to only the permissions required for development tasks. - Consider separating permissions by function or environment (e.g., dev vs. production access). - Enforce MFA for all developer accounts. - Regularly review group membership to ensure only active developers with a legitimate need are members.

3.1 Notable High-Risk Exposures

🔴 launch-wizard-6 — All Traffic Open to Internet

This security group allows all traffic from the internet. This is the most severe EC2 finding — any service running on instances associated with this group is completely exposed with no port-level filtering whatsoever.

Immediate Action Required: Remove the 0.0.0.0/0 all-traffic inbound rule immediately.

🔴 SSH (Port 22) Exposed — launch-wizard-1, testsecuritygroup

SSH exposed to the internet allows brute force attacks, credential stuffing, and exploitation of SSH vulnerabilities against these instances.

Recommendation: Restrict SSH access to known, trusted IP ranges or, preferably, remove SSH access entirely in favour of AWS Systems Manager Session Manager.

🔴 RDP (Port 3389) Exposed — launch-wizard-7

Remote Desktop Protocol exposed to the internet is a well-known attack vector heavily targeted by ransomware operators and credential brute-force attacks.

Recommendation: Remove public RDP access immediately. Use a VPN, bastion host, or SSM Session Manager for remote access.

🟠 Elasticsearch (Port 9200) and Kibana (Port 5601) Exposed — Multiple Groups

Ports 9200 (Elasticsearch API) and 5601 (Kibana) are exposed on launch-wizard-2, launch-wizard-3, launch-wizard-4, and launch-wizard-5. Unauthenticated or weakly authenticated Elasticsearch instances have historically led to mass data breaches.

Recommendation: Immediately restrict access to these ports. Elasticsearch and Kibana should never be directly accessible from the public internet. Place them behind a VPC with access only from trusted internal CIDR ranges or application-layer proxies.

🟠 Logstash (Port 5044) Exposed — launch-wizard-5

Port 5044 (Logstash Beats input) is exposed to the internet. This could allow an attacker to inject arbitrary log data into the logging pipeline, potentially poisoning log integrity.

3.2 General EC2 Security Group Recommendations

  1. Audit all security groups and remove any inbound rules sourced from 0.0.0.0/0 or ::/0 unless there is a specific, documented business requirement.
  2. Replace SSH/RDP direct access with AWS Systems Manager Session Manager, eliminating the need for any inbound SSH or RDP rules.
  3. Scope web access carefully — ports 80 and 443 may be legitimately public, but only if the instance is intended to serve public web traffic. Place other services (Elasticsearch, Kibana, Logstash) behind a private subnet.
  4. Review the "launch-wizard" naming convention — security groups with this naming pattern are typically auto-generated by the EC2 launch wizard and may not have been deliberately configured. They should be replaced with purpose-built security groups with appropriate controls.
  5. Enable VPC Flow Logs to capture network traffic metadata for security monitoring and incident response.

6. Conclusion

The AWS environment assessed presents a serious security posture risk. The complete absence of CloudTrail logging means the environment has been operating without any audit trail, making it impossible to determine whether these vulnerabilities have already been exploited. The broad IAM permissions across multiple principals and the numerous internet-exposed services compound this risk significantly.

Immediate action should be taken to: 1. Enable CloudTrail to restore visibility. 2. Close high-risk network exposure (all-traffic, SSH, RDP, Elasticsearch, Kibana). 3. Enforce least-privilege IAM policies across all identified principals.

It is recommended that a full incident response review be considered given the extended period of unmonitored, over-privileged access and internet-exposed services.
