Findings by Service

1. CloudTrail

🔴 CRITICAL — No CloudTrail Trails Found

Resource: N/A (Account-Wide) Issue: No AWS CloudTrail trails have been configured in this account. CloudTrail provides a complete audit log of all API activity across AWS services, including management console actions, SDK calls, and CLI commands.

Impact: - There is no visibility into who is performing actions within the account. - Incident investigation and forensic analysis are effectively impossible. - This finding represents a compliance violation under most regulatory frameworks (e.g., PCI-DSS, ISO 27001, SOC 2, CIS AWS Foundations Benchmark). - Malicious activity, including privilege escalation, data exfiltration, or resource deletion, would go entirely undetected and unrecorded.

Recommendation: - Immediately enable CloudTrail with a multi-region trail to capture activity across all regions. - Enable log file validation to ensure integrity of audit logs. - Store CloudTrail logs in a dedicated, access-controlled S3 bucket with versioning and MFA delete enabled. - Integrate CloudTrail with CloudWatch Logs or a SIEM solution for real-time alerting. - Ensure the trail captures both management events and, where appropriate, data events.

2. IAM — Identity & Access Management

A total of 8 IAM findings were identified, affecting IAM users, groups, and roles. The dominant theme is the assignment of overly permissive managed policies, which violates the principle of least privilege.

🟠 HIGH — ai_scanner_user — Overly Permissive Managed Policy

Resource: IAM User — ai_scanner_user Issue: This user has been assigned a managed policy that grants permissions beyond what is necessary for its function.

Recommendation: - Audit the permissions assigned to ai_scanner_user and identify which actions are actively used (via IAM Access Advisor or AWS IAM last accessed data). - Replace the overly broad managed policy with a custom least-privilege policy scoped to only the actions and resources required for the scanner's operation. - Consider using an IAM Role rather than a long-lived IAM user to reduce credential exposure risk.

🟠 HIGH — terraform-bootstrap — Overly Permissive Managed Policy

Resource: IAM User/Role — terraform-bootstrap Issue: The terraform-bootstrap resource has been assigned a managed policy that is overly permissive. Infrastructure-as-code execution roles are high-value targets due to their typically broad permissions.

Recommendation: - Scope the Terraform bootstrap role's permissions to only those required for the specific infrastructure it manages. - Implement permission boundaries to cap the maximum permissions this role can ever grant. - Use short-lived credentials via assumed roles rather than static access keys. - Review whether this role needs to remain active permanently or can be restricted to deployment pipelines only.

🟠 HIGH — Developers Group — Overly Permissive Managed Policy

Resource: IAM Group — Developers Affected Users: hughie, sasha, david, daphne, john Issue: The Developers IAM group has an overly permissive managed policy. All five members of this group inherit these excessive permissions.

Impact: - Any compromise of any one of the five developer accounts (hughie, sasha, david, daphne, john) could result in widespread access to AWS resources. - Developers may have access to production systems, sensitive data stores, or the ability to modify security configurations.

Recommendation: - Conduct a full review of the Developers group policy. - Apply role-based access control (RBAC) — differentiate between read-only, developer, and admin tiers. - Enforce MFA for all developer accounts. - Use AWS IAM Identity Center (SSO) with permission sets to manage developer access more granularly. - Remove or restrict any *:* wildcard permissions immediately.

🟠 HIGH — AWSServiceRoleForResourceExplorer — Overly Permissive Managed Policy

Resource: IAM Service-Linked Role — AWSServiceRoleForResourceExplorer Issue: This service-linked role grants access to a very large number of actions across a wide range of services, including: - 12 actions across organizations, resource-explorer-2 — all resources - 725 actions across access-analyzer, acm, acm-pca, airflow, amplify, and 183 additional services — all resources - 139 actions across backup, bedrock, bedrock-agentcore, budgets, cleanrooms, and 39 additional services — all resources

If not, disable the service to remove the role. - If Resource Explorer is needed, ensure it is scoped to the minimum required regions and resource types. - Monitor this role's activity via CloudTrail (once enabled).

🟠 HIGH — AWSServiceRoleForSupport — Overly Permissive Managed Policy

Resource: IAM Service-Linked Role — AWSServiceRoleForSupport Issue: This service-linked role carries an exceptionally large number of permissions: - 1,504 actions across access-analyzer, account, acm, acm-pca, aidevops, and 82 additional services — all resources - 1,359 actions across eks, elasticache, elasticbeanstalk, elasticfilesystem, elasticloadbalancing, and 79 additional services — all resources - 1,520 actions across oam, observabilityadmin, odb, omics, opsworks, and 90 additional services — all resources

3. EC2 — Security Groups

A total of 19 EC2 findings were identified across 7 security groups. All findings relate to inbound rules that permit unrestricted access from the internet (0.0.0.0/0 or ::/0) on sensitive ports.

🟠 HIGH — launch-wizard-4 (sg-0b1c35cdbed3ecd32) — Open Internet Access

Port Protocol Risk
80 TCP Unencrypted HTTP — potential for interception and exploitation
443 TCP HTTPS — lower direct risk but exposes service to internet
5601 TCP Kibana dashboard — unauthenticated access to log data
9200 TCP Elasticsearch API — critical; direct database access from internet

Recommendation: Immediately restrict ports 5601 and 9200. These services should never be exposed to the public internet. Place Elasticsearch and Kibana behind a VPN or bastion host. Restrict HTTP/HTTPS to known IP ranges or use an Application Load Balancer with WAF.
