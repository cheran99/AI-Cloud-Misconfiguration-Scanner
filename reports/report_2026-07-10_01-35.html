1. CloudTrail Findings

🔴 CRITICAL — No CloudTrail Trails Found

Field Detail
Service CloudTrail
Severity CRITICAL
Issue No CloudTrail trails are configured in this AWS environment

Description

AWS CloudTrail is the foundational audit and governance service for AWS accounts. Its absence means that no record exists of API calls, console logins, resource changes, or administrative actions performed in this environment. This has severe implications:

  • Incident Response is impossible — If a breach or misconfiguration occurs, there is no log data to investigate.
  • Compliance violations — Most compliance frameworks (SOC 2, PCI-DSS, ISO 27001, HIPAA, CIS AWS Foundations) mandate audit logging.
  • No detection of unauthorized access — Malicious or accidental actions leave no traceable evidence.

Remediation

  1. Enable a multi-region CloudTrail trail immediately to capture all API activity across all regions.
  2. Configure the trail to log to a dedicated, versioned, and access-controlled S3 bucket.
  3. Enable CloudTrail log file integrity validation to detect tampering.
  4. Forward logs to CloudWatch Logs for real-time alerting.
  5. Consider enabling AWS Config alongside CloudTrail for resource configuration history.

```bash aws cloudtrail create-trail \ --name management-events-trail \ --s3-bucket-name \ --is-multi-region-trail \ --enable-log-file-validation

aws cloudtrail start-logging --name management-events-trail ```

2. IAM Findings

A total of 8 IAM resources were found to have overly permissive managed policies. These findings are broken down by resource type below.

2.1 🟠 HIGH — IAM Users / Service Accounts with Overly Permissive Policies

ai_scanner_user

Field Detail
Resource Type IAM User
Severity HIGH
Issue Attached managed policy grants excessive permissions

Description: The ai_scanner_user service account has been granted an overly permissive managed policy. Service accounts should follow the principle of least privilege, with permissions scoped only to the specific actions and resources required for their function.

Remediation: - Audit the managed policies attached to ai_scanner_user. - Replace broad managed policies (e.g., AdministratorAccess, PowerUserAccess) with a custom inline or managed policy scoped to only the required actions and resources. - Consider using IAM Roles instead of long-lived IAM user credentials for service accounts.

terraform-bootstrap

Field Detail
Resource Type IAM User / Role
Severity HIGH
Issue Attached managed policy grants excessive permissions

Description: The terraform-bootstrap resource, likely used for infrastructure provisioning via Terraform, holds an overly permissive policy. Infrastructure-as-Code (IaC) execution roles are a common target for privilege escalation if compromised.

Remediation: - Scope the terraform-bootstrap permissions to only the services and actions required during deployment. - Use separate roles per environment (dev, staging, prod) with progressively restricted permissions. - Ensure Terraform state files are stored in an encrypted, access-controlled S3 bucket with state locking via DynamoDB. - Rotate or remove long-lived credentials associated with this resource.

2.2 🟠 HIGH — IAM Group with Overly Permissive Policy

Developers Group

Field Detail
Resource Type IAM Group
Severity HIGH
Issue Overly permissive managed policy applied to the Developers group
Affected Users hughie, sasha, david, daphne, john

Description: The Developers IAM group has an overly permissive managed policy, which is inherited by all 5 member users: hughie, sasha, david, daphne, and john. This represents a broad blast radius — if any one of these accounts is compromised, an attacker inherits all permissions granted to the group.

Remediation: - Review and restrict the managed policy attached to the Developers group. - Implement permission boundaries for developer users to cap the maximum permissions they can assume. - Enforce MFA for all developer accounts. - Consider role-based access via IAM Identity Center (SSO) with time-limited sessions rather than persistent IAM user credentials. - Conduct a quarterly access review for all group members.

3.1 Critical Port Exposure Details

Ports 9200 (Elasticsearch) and 5601 (Kibana)

  • Affected Groups: launch-wizard-2, launch-wizard-3, launch-wizard-4, launch-wizard-5
  • Risk: Elasticsearch and Kibana are exposed to the internet without authentication by default. Public exposure of port 9200 has historically led to mass data exfiltration events and ransomware attacks on unprotected clusters.

Port 5044 (Logstash)

  • Affected Groups: launch-wizard-5
  • Risk: Logstash input exposed to the internet could allow log injection or unauthorized data ingestion.

Port 22 (SSH)

  • Affected Groups: launch-wizard-1, testsecuritygroup
  • Risk: Publicly accessible SSH is a primary vector for brute-force attacks and unauthorized remote access.

Port 3389 (RDP)

  • Affected Groups: launch-wizard-7
  • Risk: Publicly accessible RDP is one of the most exploited attack vectors, frequently targeted by ransomware groups and credential-stuffing attacks.

All Traffic (launch-wizard-6)

  • Affected Groups: launch-wizard-6 (sg-003020051a81a8c82)
  • Risk: This is the most dangerous security group configuration possible — all protocols and all ports are exposed to the public internet. This provides zero network-level protection.

3.2 Security Group Remediation

Immediate Actions: 1. Remove all 0.0.0.0/0 inbound rules from all listed security groups immediately. 2. For SSH (22) and RDP (3389): Replace with AWS Systems Manager Session Manager to eliminate the need for open inbound ports entirely. 3. For web traffic (80/443): Place instances behind an Application Load Balancer (ALB) with WAF and restrict direct instance access. 4. For Elasticsearch (9200) and Kibana (5601): Move these services to private subnets and access via VPN or bastion/SSM only. 5. For Logstash (5044): Restrict inbound to known IP ranges of log shippers or use AWS PrivateLink. 6. Delete or rebuild launch-wizard-6 — an "all traffic" open rule serves no legitimate production purpose.

Best Practices: - Use VPC Security Group referencing rather than IP-based CIDR rules where possible. - Implement AWS Network Firewall or GuardDuty for network-level threat detection. - Remove all launch-wizard-* security groups that were auto-generated by EC2 launch wizards — these are frequently misconfigured and should be replaced with deliberately designed, named security groups.
