1. CloudTrail

🔴 CRITICAL — No Trails Found

Resource: N/A Issue: No AWS CloudTrail trails are configured in this environment.

Risk: Without CloudTrail enabled, there is no audit logging of API calls, user activity, or resource changes across the AWS account. This means: - Malicious or accidental activity cannot be detected or investigated. - Compliance requirements (e.g., PCI-DSS, SOC 2, HIPAA, ISO 27001) are likely violated. - Incident response and forensic capabilities are severely impaired.

Recommendation: - Immediately enable CloudTrail in all AWS regions. - Configure a trail to log to a dedicated, access-controlled S3 bucket with server-side encryption enabled. - Enable log file validation to detect tampering. - Consider integrating CloudTrail with CloudWatch Logs for real-time alerting on suspicious activity.

2. IAM — Overly Permissive Policies

A total of 6 IAM roles/users were found with overly permissive managed policies. Excessive permissions violate the Principle of Least Privilege and significantly increase the blast radius of a compromised credential.

🟠 HIGH — ai_scanner_user

Issue: This IAM user has an overly permissive managed policy attached.

Risk: If this user's credentials are compromised, an attacker could leverage excessive permissions to access, modify, or destroy resources across the AWS environment.

Recommendation: - Audit and scope down the attached managed policy to only the permissions required for the user's function. - Replace broad AWS managed policies (e.g., AdministratorAccess, PowerUserAccess) with custom inline or managed policies with explicit, minimal permissions. - Enforce MFA for this user and rotate access keys regularly. - Consider using IAM roles with short-lived credentials instead of long-term user access keys.

🟠 HIGH — terraform-bootstrap

Issue: This IAM entity has an overly permissive managed policy attached.

Risk: Terraform bootstrap roles are often used during infrastructure provisioning and may grant broad administrative access. If left unscoped after initial setup, they represent a persistent high-privilege attack surface.

Recommendation: - Scope this role/user's permissions down to only what is required for ongoing Terraform operations. - If this entity was only needed for initial bootstrapping, consider disabling or deleting it. - Enforce strict access controls and audit usage via CloudTrail (once enabled). - Use Terraform remote state with locked-down S3 and DynamoDB permissions.

🟠 HIGH — AWSServiceRoleForResourceExplorer

Issue: This AWS service-linked role has overly permissive managed policies, including: - 12 actions across organizations, resource-explorer-2 — Resource: all resources - 725 actions across access-analyzer, acm, acm-pca, airflow, amplify, and 183+ more services — Resource: all resources - 139 actions across backup, bedrock, bedrock-agentcore, budgets, cleanrooms, and 39+ more services — Resource: all resources

Risk: While this is an AWS-managed service-linked role, the breadth of permissions granted (725+ actions across 180+ services) goes far beyond what Resource Explorer requires to index resources. Any compromise of this role could allow broad read access to sensitive metadata across the entire account.

Recommendation: - Review whether AWS Resource Explorer is actively required. If not, disable the service to remove this role. - Monitor the role's usage via CloudTrail. - AWS does not allow direct modification of service-linked role policies; raise a support case if the permissions appear disproportionate to the service's requirements. - Apply AWS Organizations SCPs to restrict what service-linked roles can access at the organizational level.

🟠 HIGH — AWSServiceRoleForSupport

Issue: This AWS service-linked role has overly permissive managed policies, including: - 1,504 actions across access-analyzer, account, acm, acm-pca, aidevops, and 82+ more services — Resource: all resources - 1,359 actions across eks, elasticache, elasticbeanstalk, elasticfilesystem, elasticloadbalancing, and 79+ more services — Resource: all resources - 1,520 actions across oam, observabilityadmin, odb, omics, opsworks, and 90+ more services — Resource: all resources

Risk: This role grants AWS Support engineers access to your account for troubleshooting. With over 4,300+ actions across the majority of AWS services, this role has near-complete read (and in some cases write) access to the environment. If misused or accessed inappropriately, sensitive data and configurations could be exposed.

Recommendation: - Review your AWS Support tier. If you are on Basic Support and do not require AWS support access, evaluate whether this service-linked role's access can be limited. - Regularly review AWS Support cases to ensure no unauthorized cases are opened. - Leverage AWS Organizations SCPs to limit the scope of what Support can access where possible. - Enable AWS Config and CloudTrail to monitor any activity performed under this role.

🟠 HIGH — AWSServiceRoleForTrustedAdvisor

Issue: This AWS service-linked role has an overly permissive managed policy, including: - 115 actions across access-analyzer, autoscaling, ce, cloudformation, cloudfront, and 19+ more services — Resource: all resources

Risk: Trusted Advisor requires broad read access to evaluate account health. However, the wide resource scope (all resources) means any exploitation of this role could expose configuration and cost data across the entire account.

Recommendation: - Ensure Trusted Advisor is genuinely required and in use. - Monitor activity under this role via CloudTrail. - As a service-linked role, policies cannot be directly modified; apply SCPs at the organizational level to enforce boundaries.

launch-wizard-4 (sg-0b1c35cdbed3ecd32) Open ports: 80 (HTTP), 443 (HTTPS), 5601 (Kibana), 9200 (Elasticsearch)

Exposing Kibana (5601) and Elasticsearch (9200) directly to the internet is particularly dangerous, as these services have historically been targeted in mass data exfiltration attacks due to lack of authentication by default.

launch-wizard-2 (sg-0a0df35eef090ae38) Open ports: 80 (HTTP), 5601 (Kibana), 9200 (Elasticsearch)

Same Kibana/Elasticsearch exposure as above. The absence of TLS (no port 443) compounds the risk.

launch-wizard-5 (sg-086216d42764aa08e) Open ports: 443 (HTTPS), 5044 (Logstash Beats), 5601 (Kibana), 9200 (Elasticsearch)

This security group appears to expose a full ELK/Elastic Stack to the internet (Logstash, Kibana, Elasticsearch). This is a critical data exposure risk and likely indicates a logging infrastructure instance directly accessible from the internet.

launch-wizard-1 (sg-0ddc1c39a3fcc2123) Open ports: 22 (SSH), 443 (HTTPS)

SSH (port 22) open to the internet exposes the instance to brute-force attacks, credential stuffing, and exploitation of SSH vulnerabilities. This is a common initial access vector.

launch-wizard-6 (sg-003020051a81a8c82) Open ports: All Traffic

This is the most permissive security group in the environment. Allowing all traffic from the internet (0.0.0.0/0) on all protocols and ports provides absolutely no network-level protection. This must be treated as a critical misconfiguration.

launch-wizard-7 (sg-0c3704fd44e21ac07) Open ports: 3389 (RDP)

RDP exposed to the internet is a well-known and frequently exploited attack vector, associated with ransomware deployment and lateral movement. This should be remediated immediately.

testsecuritygroup (sg-0b2da4349ee7bc901) Open ports: 22 (SSH)

Even "test" security groups represent a real risk if attached to active instances. SSH open to the internet is a common initial access vector.

EC2 Security Group Recommendations

  1. Immediately restrict all inbound rules scoped to 0.0.0.0/0 on sensitive ports.
  2. For SSH (22) and RDP (3389): Remove public access entirely. Use AWS Systems Manager Session Manager or EC2 Instance Connect with IP-restricted access instead.
  3. For Elasticsearch (9200) and Kibana (5601): These should never be publicly accessible. Place them behind a VPC with private subnets, and use a bastion host or VPN for access.
  4. For Logstash (5044): Restrict to known log-shipping source IPs only.
  5. For HTTP/HTTPS (80/443): If web access is required, use an Application Load Balancer (ALB) with WAF in front, not direct instance exposure.
  6. For launch-wizard-6 (All Traffic): Audit immediately. Remove this security group from any attached instances and replace with a scoped policy.
  7. Use AWS Config Rules (e.g., restricted-ssh, restricted-common-ports) to automatically detect and alert on open security group rules going forward.
  8. Audit all "launch-wizard" security groups — these are auto-generated by the EC2 launch wizard and often left with overly permissive rules. Implement a naming convention and security group lifecycle management policy.

General Recommendations

  1. Enable AWS Security Hub to consolidate findings from GuardDuty, Inspector, Macie, and Config into a single dashboard.
  2. Enable Amazon GuardDuty for intelligent threat detection across CloudTrail, VPC Flow Logs, and DNS.
  3. Implement VPC Flow Logs to capture network traffic data for analysis and incident response.
  4. Enforce a least-privilege IAM strategy using AWS IAM Access Analyzer to identify and remediate overly permissive policies continuously.
  5. Adopt AWS Organizations SCPs to enforce guardrails across the entire account or organization.
  6. Establish a Security Group Review process — prohibit auto-generated "launch-wizard" security groups in production and enforce tagging and lifecycle management.
  7. Enable MFA on all IAM users, especially those with elevated privileges.
