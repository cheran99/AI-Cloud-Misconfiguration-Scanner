Critical Finding

⛔ CloudTrail — No Audit Logging Configured

Field Detail
Severity CRITICAL
Service CloudTrail
Resource N/A

Description: No AWS CloudTrail trails were found in this environment. CloudTrail is the primary mechanism for auditing API activity across all AWS services. Without it, there is no record of who did what, when, and from where — making incident detection, forensic investigation, and compliance reporting impossible.

Risk: - Inability to detect malicious or accidental changes - Failure to meet compliance requirements (PCI-DSS, ISO 27001, SOC 2, GDPR, etc.) - No audit trail for insider threats or compromised credentials

Remediation: 1. Enable at least one multi-region CloudTrail trail immediately. 2. Configure log delivery to a dedicated, versioned, and MFA-delete-protected S3 bucket. 3. Enable CloudTrail log file integrity validation. 4. Consider integrating with AWS CloudWatch Logs and AWS Security Hub for alerting.

High Severity Findings

🟠 IAM — Overly Permissive IAM Policies

Overview

Seven IAM findings were identified across multiple roles and users. Overly permissive policies — particularly those granting access to all resources (*) — violate the principle of least privilege and represent a significant blast radius if credentials are compromised.

Finding 1 — ai_scanner_user

Field Detail
Severity HIGH
Service IAM
Resource ai_scanner_user

Description: The IAM user ai_scanner_user has been assigned an overly permissive managed policy. IAM users with broad permissions pose a heightened risk as they rely on long-lived static credentials (access keys), which can be leaked or stolen.

Remediation: - Audit and scope the attached managed policy to only the actions and resources required for the user's function. - Replace static IAM user credentials with IAM roles where possible. - Enable MFA on the user account. - Rotate and audit access keys regularly.

Finding 2 — terraform-bootstrap

Field Detail
Severity HIGH
Service IAM
Resource terraform-bootstrap

Description: The terraform-bootstrap IAM entity has an overly permissive managed policy. Terraform provisioning roles commonly require broad permissions during setup but should be locked down post-bootstrap or scoped using permission boundaries.

Remediation: - Review whether the terraform-bootstrap role is still in active use; remove if not. - If still required, apply an IAM permissions boundary to limit effective permissions. - Restrict the policy to only the actions needed for current Terraform operations. - Use Service Control Policies (SCPs) to enforce guardrails at the organisation level.

Finding 3 — AWSServiceRoleForResourceExplorer

Field Detail
Severity HIGH
Service IAM
Resource AWSServiceRoleForResourceExplorer

Description: This AWS-managed service-linked role has been flagged for granting a large number of actions across a wide range of services on all resources: - 12 actions across organizations, resource-explorer-2 - 725 actions across 188+ services (including access-analyzer, acm, amplify, and more) - 139 actions across 44+ services (including backup, bedrock, budgets, and more)

Remediation: - Evaluate whether AWS Resource Explorer is required in this environment. - If not in use, disable Resource Explorer to remove this role and its associated permissions. - If in use, ensure access to the Resource Explorer console is restricted to authorised personnel via IAM identity policies.

🟠 S3 — Insufficient Encryption

Finding — dev-project-assets-211125321299-eu-west-2-an

Field Detail
Severity HIGH
Service S3
Resource dev-project-assets-211125321299-eu-west-2-an

Description: The S3 bucket dev-project-assets-211125321299-eu-west-2-an is not encrypted using AWS KMS (SSE-KMS). While the bucket may use SSE-S3 (AES-256), this does not provide the same level of control, auditability, or key management as SSE-KMS.

Risk: - Inability to audit data access at the encryption key level via AWS KMS CloudTrail events. - No separation of key management duties from storage administration. - May not meet compliance requirements that mandate customer-managed key (CMK) encryption.

Remediation: 1. Create or identify an appropriate AWS KMS Customer Managed Key (CMK). 2. Update the bucket's default encryption configuration to use SSE-KMS with the designated CMK. 3. Apply a bucket policy to deny uploads that do not use SSE-KMS (s3:x-amz-server-side-encryption condition). 4. Review existing objects in the bucket and re-encrypt using S3 Batch Operations if required. 5. Ensure the KMS key policy follows the principle of least privilege.

🟠 EC2 — Security Groups with Open Internet Access

Overview

Nineteen findings were identified across seven security groups, all exposing various ports to the public internet (0.0.0.0/0 or ::/0). Open security groups are one of the most common causes of AWS breaches and can expose sensitive services directly to attackers.

Security Group: launch-wizard-4 (sg-0b1c35cdbed3ecd32)

Port Protocol Exposure

443 TCP Open to Internet
5601 TCP Open to Internet (Kibana)

Risk: Ports 5601 (Kibana) and 9200 (Elasticsearch) are particularly dangerous when exposed publicly. Unauthenticated Elasticsearch instances have been responsible for numerous large-scale data breaches.

Risk: This is the most dangerous configuration identified. All ports and all protocols are open to the entire internet, exposing every service running on associated instances to potential attack.

3389 TCP Open to Internet (RDP)

Risk: Exposing RDP (port 3389) to the internet is a well-known attack vector for brute-force attacks, credential stuffing, and exploitation of RDP vulnerabilities (e.g., BlueKeep).

Security Group: launch-wizard-1 (sg-0ddc1c39a3fcc2123)

Port Protocol Exposure
22 TCP Open to Internet (SSH)
443 TCP Open to Internet

Risk: Port 22 (SSH) open to the internet is a primary target for brute-force and credential-stuffing attacks.

Security Group: testsecuritygroup (sg-0b2da4349ee7bc901)

Port Protocol Exposure
22 TCP Open to Internet (SSH)

Risk: Even test security groups with open SSH access pose a risk if attached to any live instance. Test resources are frequently left running unintentionally.

EC2 Security Group Remediation (All Groups)

  1. Remove all inbound rules that allow 0.0.0.0/0 or ::/0 on sensitive ports (22, 3389, 9200, 5601, 5044).
  2. Restrict SSH (22) and RDP (3389) to specific trusted IP ranges or, preferably, eliminate direct access entirely in favour of:
  3. AWS Systems Manager Session Manager for SSH/RDP-equivalent access without open ports.
  4. EC2 Instance Connect scoped to specific IPs.
  5. Restrict Elasticsearch (9200) and Kibana (5601) to internal VPC CIDR ranges only. Place these behind a VPN or AWS PrivateLink.
  6. Restrict Logstash (5044) to known log shipper IP addresses or use VPN/PrivateLink.
  7. Remove launch-wizard-6 open-all-traffic rule immediately. This is critical.
  8. Use AWS VPC Security Group best practices: default-deny with explicit allow-list rules per service.
  9. Enable AWS VPC Flow Logs to monitor traffic patterns and detect anomalous access.
  10. Use AWS Config rules (e.g., restricted-ssh, restricted-common-ports) to automatically detect and alert on permissive security groups.
  11. Rename and document security groups — launch-wizard-* names indicate these were created via the EC2 launch wizard without deliberate naming, suggesting ad-hoc provisioning practices.

Conclusion

The environment presents a significant security posture risk driven primarily by:

  1. Complete absence of audit logging — the single most critical gap, eliminating any visibility into account activity.
  2. Widespread open security groups — multiple services including Elasticsearch, Kibana, SSH, RDP, and Logstash are directly exposed to the internet.
  3. Overly permissive IAM policies — both user-managed and service-linked roles hold far broader permissions than necessary.
  4. Insufficient data encryption controls — S3 assets lack KMS-based encryption.

Immediate action is required on the items identified above. It is strongly recommended that a broader security review including penetration testing, AWS Security Hub enablement, and a formal IAM access review be initiated in parallel with remediation activities.
