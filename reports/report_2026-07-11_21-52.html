2. IAM — Overly Permissive Policies

Multiple IAM roles and users have been granted overly broad permissions, violating the Principle of Least Privilege (PoLP). Overly permissive policies increase the blast radius of any credential compromise.

🟠 HIGH — ai_scanner_user: Overly Permissive Managed Policy

Issue: The IAM user ai_scanner_user has been assigned a managed policy that grants excessive permissions.

Recommendation: - Audit the policies attached to ai_scanner_user. - Restrict permissions to only those actions explicitly required for its function. - Replace broad managed policies with a custom, scoped-down inline or managed policy. - Enforce regular access reviews.

🟠 HIGH — terraform-bootstrap: Overly Permissive Managed Policy

Issue: The IAM role/user terraform-bootstrap has an overly permissive managed policy attached.

Recommendation: - Terraform provisioning roles often require broad permissions during initial setup but should be tightly scoped post-bootstrapping. - Review whether this role is still in active use. If not, disable or delete it. - If still required, scope permissions down to only the specific services and actions needed. - Consider using temporary credentials via AWS IAM Identity Center rather than long-lived access keys.

🟠 HIGH — AWSServiceRoleForResourceExplorer: Overly Permissive Managed Policy

Issue: This AWS-managed service-linked role has policies granting: - 12 actions across organizations, resource-explorer-2 — all resources - 725 actions across access-analyzer, acm, acm-pca, airflow, amplify, and 183 additional services — all resources - 139 actions across backup, bedrock, bedrock-agentcore, budgets, cleanrooms, and 39 additional services — all resources

Context: This is an AWS-managed service-linked role. The breadth of permissions reflects the read-access requirements of Resource Explorer to index resources across the account.

Recommendation: - Verify that AWS Resource Explorer is a required and actively used service. If not, disable it to remove this role. - Confirm the role has not been modified beyond its AWS-managed defaults. - Periodically review AWS service-linked role policies as AWS updates them.

🟠 HIGH — AWSServiceRoleForSupport: Overly Permissive Managed Policy

Issue: This AWS-managed service-linked role has policies granting: - 1,504 actions across access-analyzer, account, acm, acm-pca, aidevops, and 82 additional services — all resources - 1,359 actions across eks, elasticache, elasticbeanstalk, elasticfilesystem, elasticloadbalancing, and 79 additional services — all resources - 1,520 actions across oam, observabilityadmin, odb, omics, opsworks, and 90 additional services — all resources

Recommendation: - Ensure this role has not been modified beyond AWS defaults. - If AWS Support is not actively being used, consider reviewing your AWS Support plan and whether this service-linked role can be removed. - Monitor any actions taken under this role via CloudTrail (once enabled).

🟠 HIGH — AWSServiceRoleForTrustedAdvisor: Overly Permissive Managed Policy

🟠 HIGH — EC2InstanceConnectRole: Overly Permissive Managed Policy (×2 findings)

Issue: Two separate policy findings were identified for this role:

Finding 1: - 1 action across ec2-instance-connect — all resources

Finding 2: - 14 actions across ssm — all resources - 4 actions across ssmmessages — all resources - 6 actions across ec2messages — all resources - 1 action across cloudwatch — all resources - 1 action across ec2 — all resources - 2 actions across ds — all resources - 5 actions across logs — all resources - 8 actions across s3 — all resources

Impact: The role grants access to all resources rather than scoping permissions to specific EC2 instances or S3 buckets. If compromised, this role could be used to connect to any EC2 instance and access unintended S3 data.

🟠 HIGH — Elasticsearch & Kibana Stack Exposed (Ports 9200, 5601, 5044)

Affected Groups: launch-wizard-2, launch-wizard-3, launch-wizard-4, launch-wizard-5

Issue: Ports 9200 (Elasticsearch REST API) and 5601 (Kibana) are open to the internet. Port 5044 (Logstash) is also exposed on launch-wizard-5.

Impact: - Elasticsearch instances without authentication are a well-known attack vector and have been responsible for numerous large-scale data breaches. - An attacker can read, modify, or delete all indexed data. - Kibana exposure allows direct interface access to the Elasticsearch cluster.

Recommendation: - Immediately restrict these ports to specific trusted IP ranges or a VPN/bastion host. - Enable Elasticsearch security features (TLS, authentication) if not already configured. - Place the Elasticsearch cluster in a private subnet with no direct internet routing.

🟠 HIGH — SSH Open to Internet (Port 22)

Affected Groups: launch-wizard-1 (sg-0ddc1c39a3fcc2123), testsecuritygroup (sg-0b2da4349ee7bc901)

Issue: Port 22 (SSH) is open to 0.0.0.0/0.

Impact: - Exposes instances to brute-force and credential-stuffing attacks. - A single compromised SSH key or weak password could result in full server access.

Recommendation: - Remove the 0.0.0.0/0 inbound rule for port 22 immediately. - Restrict SSH access to specific corporate IP ranges or a VPN CIDR. - Adopt EC2 Instance Connect or AWS Systems Manager Session Manager as a replacement for direct SSH access — eliminating the need to expose port 22 entirely.

🟠 HIGH — RDP Open to Internet (Port 3389)

Affected Groups: launch-wizard-7 (sg-0c3704fd44e21ac07)

Issue: Port 3389 (RDP) is open to 0.0.0.0/0.

Impact: - RDP is a frequent target for brute-force attacks and ransomware campaigns. - BlueKeep and related vulnerabilities have historically enabled unauthenticated remote code execution via exposed RDP.

Recommendation: - Immediately remove the open RDP rule. - Restrict to specific trusted IP addresses or VPN access only. - Consider using AWS Systems Manager Fleet Manager for RDP access without exposing port 3389.

🟠 HIGH — All Traffic Open to Internet

Affected Groups: launch-wizard-6 (sg-003020051a81a8c82)

Issue: All inbound traffic from 0.0.0.0/0 is permitted — across all protocols and all ports.

Impact: - This is the most severe security group misconfiguration possible. Every port on any instance using this security group is reachable from the public internet. - Any service running on the instance, including administrative interfaces, databases, and internal APIs, is exposed.

Recommendation: - Immediately remove this rule. - Replace with tightly scoped rules permitting only the specific ports and protocols required. - Audit all instances currently associated with this security group.

🟠 HIGH — HTTP/HTTPS Open to Internet (Ports 80, 443)

Affected Groups: launch-wizard-1, launch-wizard-3, launch-wizard-4, launch-wizard-5

Issue: Ports 80 (HTTP) and 443 (HTTPS) are open to 0.0.0.0/0.

Context: Web access may be intentional for public-facing applications; however, given the broader exposure of Elasticsearch and Kibana on the same security groups, this warrants review.

Recommendation: - Confirm whether public web access is intentional. - If serving web traffic publicly, route it through an AWS Application Load Balancer (ALB) with AWS WAF enabled rather than directly to EC2 instances. - Ensure HTTP (port 80) redirects to HTTPS and is not serving sensitive content unencrypted.

4. S3 — Insufficient Encryption

🟠 HIGH — dev-project-assets-211125321299-eu-west-2-an: Encryption Not Using AWS KMS

Issue: The S3 bucket is not encrypted using AWS KMS (Key Management Service). It is likely using SSE-S3 (AES-256 managed by Amazon) or has no encryption configured.

Impact: - Without KMS encryption, there is no ability to enforce fine-grained key access controls, audit key usage, or rotate encryption keys on a defined schedule. - Fails compliance requirements for frameworks requiring customer-managed key (CMK) encryption (PCI-DSS, HIPAA, SOC 2, ISO 27001).

Recommendation: - Enable SSE-KMS on the bucket using either an AWS-managed key (aws/s3) or, preferably, a Customer Managed Key (CMK). - Apply a bucket policy that denies PutObject requests that do not specify KMS encryption (s3:x-amz-server-side-encryption: aws:kms). - Enable KMS key rotation for the CMK.
