Detailed Findings

1. CloudTrail

🔴 CRITICAL — No CloudTrail Trails Found

Description: No AWS CloudTrail trails have been configured in this environment. CloudTrail is the primary mechanism for logging API activity across AWS services. Without it, there is no audit trail of user actions, resource changes, or API calls, making it impossible to detect, investigate, or respond to security incidents.

Impact: - Complete loss of visibility into account activity. - Inability to perform forensic analysis following a security incident. - Non-compliance with regulatory frameworks including PCI-DSS, HIPAA, SOC 2, ISO 27001, and CIS AWS Benchmarks. - Potential failure to detect unauthorized access or insider threats.

Recommended Remediation: 1. Immediately create a CloudTrail trail covering all regions. 2. Enable log file integrity validation to detect tampering. 3. Store logs in a dedicated, access-restricted S3 bucket with server-side encryption (SSE-KMS). 4. Enable CloudWatch Logs integration for real-time alerting on suspicious activity. 5. Consider enabling AWS Config alongside CloudTrail for resource configuration tracking.

2. IAM — Overly Permissive Policies

🟠 HIGH — ai_scanner_user: Overly Permissive Managed Policy

Description: The IAM user ai_scanner_user has been granted an overly permissive managed policy. This violates the principle of least privilege, potentially granting this user far more access than required for its intended function.

Impact: - If compromised, an attacker could exploit excess permissions to escalate privileges or exfiltrate data. - Increases the blast radius of any credential compromise.

Recommended Remediation: 1. Audit all policies attached to ai_scanner_user and identify unnecessary permissions. 2. Replace broad managed policies with a custom, scoped-down policy granting only the minimum permissions required. 3. Enable IAM Access Analyzer to identify unused permissions. 4. Rotate access keys regularly and enforce MFA.

🟠 HIGH — terraform-bootstrap: Overly Permissive Managed Policy

Description: The IAM entity terraform-bootstrap has been granted an overly permissive managed policy. Terraform bootstrap roles frequently carry broad permissions due to infrastructure provisioning requirements, but these must be carefully scoped and monitored.

Impact: - If Terraform credentials are leaked (e.g., through a CI/CD pipeline misconfiguration), an attacker gains wide-ranging AWS access. - Broad permissions on automation roles can lead to unintended infrastructure changes.

Recommended Remediation: 1. Review and restrict the managed policies attached to terraform-bootstrap. 2. Where possible, use permission boundaries to cap the maximum effective permissions. 3. Restrict this role's usage to specific pipelines using condition keys (e.g., aws:SourceArn, aws:PrincipalTag). 4. Avoid long-lived credentials; use IAM roles with short-lived session tokens instead.

🟠 HIGH — AWSServiceRoleForResourceExplorer: Overly Permissive Managed Policy

Description: The AWS Service-Linked Role AWSServiceRoleForResourceExplorer has been flagged for overly permissive policies, including: - 12 actions across organizations and resource-explorer-2 on all resources. - 725 actions across 188+ services (including access-analyzer, acm, airflow, amplify, and others) on all resources. - 139 actions across 44+ services (including backup, bedrock, budgets, cleanrooms, and others) on all resources.

Impact: - While service-linked roles are AWS-managed, their broad scope warrants monitoring. - Any misconfiguration or exploitation of Resource Explorer could expose resource metadata across the entire account.

Recommended Remediation: 1. Verify that AWS Resource Explorer is actually required and actively used in the environment. 2. If not needed, disable the Resource Explorer service to remove the associated service-linked role. 3. Monitor activity associated with this role via CloudTrail (once enabled). 4. Review AWS's documentation on this role to understand scope and apply Service Control Policies (SCPs) at the organizational level if applicable.

🟠 HIGH — AWSServiceRoleForSupport: Overly Permissive Managed Policy

Description: The AWSServiceRoleForSupport role carries extremely broad permissions across hundreds of AWS services, including: - 1,504 actions across 87+ services (e.g., access-analyzer, account, acm, eks, and others). - 1,359 actions across 84+ services (e.g., eks, elasticache, elasticbeanstalk, and others). - 1,520 actions across 95+ services (e.g., oam, odb, omics, opsworks, and others).

Description: This is an AWS-managed service-linked role for AWS Support. Its broad read permissions are by design to allow Support engineers to diagnose issues.

Impact: - Exploitation of this role could expose sensitive resource configurations across the entire AWS account. - If AWS Support access is not actively needed, this represents unnecessary risk exposure.

Recommended Remediation: 1. Confirm whether AWS Support (Business or Enterprise tier) is actively subscribed to and required. 2. If not required, evaluate whether the associated support plan can be downgraded to reduce this role's presence. 3. Apply SCPs to limit what services can be accessed via this role at the organization level. 4. Audit and monitor usage of this role via CloudTrail.

🟠 HIGH — AWSServiceRoleForTrustedAdvisor: Overly Permissive Managed Policy

Description: The AWSServiceRoleForTrustedAdvisor role has broad read permissions including: - 115 actions across 24+ services (e.g., access-analyzer, autoscaling, cloudformation, cloudfront, and others) on all resources.

Impact: - While primarily read-only, broad access across services could expose sensitive configuration details.

Recommended Remediation: 1. Confirm that AWS Trusted Advisor is actively used for environment optimization. 2. If not in use, consider disabling Trusted Advisor to remove the role. 3. Monitor Trusted Advisor role activity via CloudTrail.

🟠 HIGH — EC2InstanceConnectRole: Overly Permissive Managed Policies (x2)

Description: The EC2InstanceConnectRole has been flagged across two separate policy findings:

Finding 1: - 1 action on ec2-instance-connect applied to all resources (should be scoped to specific instances).

Finding 2: - 14 actions on ssm (all resources) - 4 actions on ssmmessages (all resources) - 6 actions on ec2messages (all resources) - 1 action on cloudwatch (all resources) - 1 action on ec2 (all resources) - 2 actions on ds (all resources) - 5 actions on logs (all resources) - 8 actions on s3 (all resources)

Impact: - The role grants access to all EC2 instances rather than specific targets. - SSM, S3, and CloudWatch permissions scoped to all resources could be leveraged laterally if this role is compromised.

Recommended Remediation: 1. Restrict the ec2-instance-connect:SendSSHPublicKey action to specific instance ARNs and OS users using resource-level conditions. 2. Scope SSM, S3, and other permissions to specific resource ARNs where possible. 3. Use IAM condition keys such as ec2:ResourceTag to limit access to tagged instances. 4. Review whether all listed services are genuinely required for this role's function.

3. S3

🟠 HIGH — dev-project-assets-211125321299-eu-west-2-an: Bucket Not Encrypted with AWS KMS

Resource: dev-project-assets-211125321299-eu-west-2-an

Description: The S3 bucket is either unencrypted or using SSE-S3 (AES-256) rather than AWS KMS (SSE-KMS). KMS-based encryption provides enhanced key management capabilities, audit logging via CloudTrail, and stronger access controls.

Impact: - Data at rest may not meet organisational or regulatory encryption requirements. - Without KMS, there is no centralized key management, rotation policy enforcement, or granular access control over the encryption keys. - Non-compliant with frameworks such as PCI-DSS, HIPAA, and ISO 27001 that mandate strong encryption key management.

Recommended Remediation: 1. Enable SSE-KMS encryption on the bucket using either an AWS-managed key (aws/s3) or a customer-managed KMS key (CMK) — the latter is strongly preferred. 2. Create a dedicated KMS CMK for S3 with a defined key policy restricting access to authorized principals only. 3. Enable key rotation on the CMK (automatic annual rotation). 4. Apply a bucket policy to deny s3:PutObject requests that do not include x-amz-server-side-encryption: aws:kms. 5. Consider enabling S3 default encryption to enforce KMS on all future object uploads.

Security Group Recommended Remediation (All Groups)

  1. Immediately remove all 0.0.0.0/0 and ::/0 inbound rules on non-public-facing ports, particularly:
  2. Port 22 (SSH)
  3. Port 3389 (RDP)
  4. Port 9200 (Elasticsearch)
  5. Port 5601 (Kibana)
  6. Port 5044 (Logstash)
  7. All traffic rules.
  8. Replace unrestricted rules with rules scoped to known, trusted IP CIDR ranges (e.g., corporate VPN, office IPs).
  9. For SSH and RDP access: implement AWS Systems Manager Session Manager as a replacement to eliminate the need for open SSH/RDP ports entirely.
  10. For Elasticsearch and Kibana: place these behind a private subnet with access controlled via a VPN or bastion host. Never expose these services directly to the internet.
  11. For legitimate public web services (ports 80/443): consider placing instances behind an Application Load Balancer (ALB) and restrict direct instance security group access.
  12. Implement AWS Network Firewall or VPC Flow Logs for ongoing network traffic monitoring.
  13. Rename security groups from launch-wizard-* naming to descriptive names reflecting their purpose and enforce tagging policies.
  14. Review and remove the launch-wizard-6 all-traffic rule as an emergency priority.

Conclusion

The AWS environment assessed presents a high overall security risk profile. The absence of CloudTrail logging is the most critical finding, as it undermines the ability to detect and respond to any of the other vulnerabilities identified. The widespread use of overly permissive security groups, especially those exposing Elasticsearch, Kibana, SSH, RDP, and all-traffic rules to the public internet, creates significant risk of data breach and system compromise.

Immediate remediation of the CloudTrail gap and network-level exposures is strongly recommended, followed by a structured programme to enforce IAM least privilege and strengthen encryption practices.
