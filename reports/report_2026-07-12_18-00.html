1. CloudTrail Findings

Finding: No CloudTrail Trails Configured

Field Detail
Severity 🔴 CRITICAL
Service CloudTrail
Resource N/A — Account-wide
Issue No CloudTrail trails were found in this AWS account

Description

AWS CloudTrail is the primary mechanism for auditing API activity across an AWS account. Its absence means that no record exists of any action taken — including resource creation, deletion, permission changes, or login events. This is a critical gap in both security and compliance posture.

Impact

  • No forensic capability: In the event of a breach or misconfiguration, there is no audit log to investigate.
  • Compliance violations: CloudTrail is a mandatory control under frameworks including CIS AWS Foundations Benchmark, PCI-DSS, SOC 2, ISO 27001, and HIPAA.
  • Blind spots for threat detection: Security tools that rely on CloudTrail events (e.g., AWS GuardDuty, AWS Security Hub, SIEM integrations) cannot function effectively.

Remediation

  1. Enable CloudTrail with a multi-region trail to capture all activity across all regions.
  2. Enable log file validation to ensure trail integrity.
  3. Store logs in a dedicated, access-controlled S3 bucket with KMS encryption.
  4. Enable CloudWatch Logs integration for real-time alerting.
  5. Consider enabling AWS CloudTrail Insights to detect anomalous API activity.

```bash aws cloudtrail create-trail \ --name organisation-audit-trail \ --s3-bucket-name \ --is-multi-region-trail \ --enable-log-file-validation

aws cloudtrail start-logging --name organisation-audit-trail ```

Finding 2.1: ai_scanner_user — Overly Permissive Managed Policy

Field Detail
Severity 🟠 HIGH
Service IAM
Resource ai_scanner_user
Issue IAM user has an overly permissive managed policy attached

Description

The ai_scanner_user IAM user has been granted permissions that exceed what is necessary for its intended function. This violates the principle of least privilege and increases the blast radius if this user's credentials are compromised.

Remediation

  • Review the policies attached to ai_scanner_user and identify all actions granted.
  • Replace broad managed policies with a custom, scoped IAM policy that grants only the specific actions and resources required.
  • Ensure this user uses IAM access keys with rotation enforced, or consider migrating to IAM roles if the user represents an application workload.

3. S3 Findings

A total of 3 HIGH severity S3 findings were identified across 2 buckets.

Finding 3.1: dev-project-assets-211125321299-eu-west-2-an — Bucket Not Encrypted with KMS

Field Detail
Severity 🟠 HIGH
Service S3
Resource dev-project-assets-211125321299-eu-west-2-an
Issue Bucket encryption is not using AWS KMS

Description

The bucket is either unencrypted or using Amazon S3-managed keys (SSE-S3) rather than AWS Key Management Service (SSE-KMS). KMS encryption provides centralised key management, audit logging via CloudTrail, and the ability to enforce access controls at the key level.

Remediation

  1. Enable SSE-KMS encryption on the bucket using a customer-managed key (CMK).
  2. Create a KMS key with an appropriate key policy restricting access to authorised principals only.
  3. Enable KMS key rotation.

bash aws s3api put-bucket-encryption \ --bucket dev-project-assets-211125321299-eu-west-2-an \ --server-side-encryption-configuration '{ "Rules": [{ "ApplyServerSideEncryptionByDefault": { "SSEAlgorithm": "aws:kms", "KMSMasterKeyID": "<your-kms-key-arn>" }, "BucketKeyEnabled": true }] }'

Finding 3.2: drone-footage-dev-211125321299-eu-west-2-an — Public Access Block Not Configured

Field Detail
Severity 🟠 HIGH
Service S3
Resource drone-footage-dev-211125321299-eu-west-2-an
Issue S3 Public Access Block settings are not configured

Description

The S3 Public Access Block feature is not enabled on this bucket. Without this configuration, the bucket — and objects within it — could be made publicly accessible via bucket policies or ACLs, either intentionally or through misconfiguration. Given this bucket contains drone footage, exposure of its contents could have significant privacy and legal implications.

Remediation

Enable all four Public Access Block settings immediately:

bash aws s3api put-public-access-block \ --bucket drone-footage-dev-211125321299-eu-west-2-an \ --public-access-block-configuration \ "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

Finding 3.3: drone-footage-dev-211125321299-eu-west-2-an — Bucket Not Encrypted with KMS

Field Detail
Severity 🟠 HIGH

Resource drone-footage-dev-211125321299-eu-west-2-an

In addition to the public access finding, this bucket also lacks KMS-based encryption. Drone footage may contain sensitive or personally identifiable information (PII), making encryption at rest a critical requirement.

Issue All traffic is open to the internet

Description

This security group permits all inbound traffic on all ports and protocols from any source. This is the most dangerous security group configuration possible and effectively removes all network-layer protection from any associated resources.

Remediation

  • Immediately restrict or delete this rule.
  • Replace with rules that permit only the specific ports and source IP ranges required.

Finding 4.2: SSH/RDP Open to Internet (testsecuritygroup, launch-wizard-1, launch-wizard-7)

Port Security Group ID
TCP 22 (SSH) testsecuritygroup sg-0b2da4349ee7bc901
TCP 22 (SSH) launch-wizard-1 sg-0ddc1c39a3fcc2123
TCP 3389 (RDP) launch-wizard-7 sg-0c3704fd44e21ac07

Description

SSH and RDP exposed to the public internet are among the most commonly exploited attack vectors. Automated scanning tools continuously probe public IP space for open SSH/RDP, and exposed services are subject to brute-force, credential stuffing, and exploitation of unpatched vulnerabilities.

Remediation

  • Remove public SSH/RDP rules immediately.
  • Use AWS Systems Manager Session Manager for shell access without requiring open inbound ports.
  • If direct SSH/RDP is required, restrict the source to specific known IP addresses or CIDR ranges (e.g., a corporate VPN egress IP).
  • Consider deploying a bastion host or AWS Client VPN as a more controlled access mechanism.

Finding 4.3: Elasticsearch & Kibana Open to Internet (Multiple Security Groups)

Affected groups: launch-wizard-4, launch-wizard-2, launch-wizard-3, launch-wizard-5

Port Service
9200 Elasticsearch HTTP API
5601 Kibana
5044 Logstash Beats input

Description

Elasticsearch (port 9200) and Kibana (port 5601) open to the internet present severe data exposure risk. Historically, publicly exposed Elasticsearch instances have been the source of numerous large-scale data breaches. Without authentication (which is not enabled by default in older Elasticsearch versions), any actor can read, modify, or delete all indexed data.

Remediation

  • Immediately restrict ports 9200, 5601, and 5044 to private IP ranges or known source IPs only.
  • Ensure Elasticsearch and Kibana have authentication and TLS enabled.
  • Consider deploying Elasticsearch within a VPC with no public IP, accessible only via private networking.
  • Use AWS OpenSearch Service instead of self-managed Elasticsearch for better-integrated security controls.
