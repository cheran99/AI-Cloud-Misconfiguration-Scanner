1. Critical Findings

🔴 CRIT-001 — No AWS CloudTrail Trails Found

Field Detail
Service CloudTrail
Severity CRITICAL
Resource Account-wide

Description: No AWS CloudTrail trails exist in this AWS account. CloudTrail is the primary auditing and logging mechanism for AWS API activity. Without it, there is no record of any action taken against AWS resources — including IAM changes, EC2 modifications, data access events, or login activity.

Impact: - Complete loss of audit trail and forensic capability - Inability to detect or investigate security incidents, data breaches, or insider threats - Non-compliance with virtually all major regulatory frameworks (PCI-DSS, HIPAA, SOC 2, ISO 27001, CIS AWS Benchmark) - No alerting capability for suspicious API calls or privilege escalation events

Recommended Action: Enable a multi-region CloudTrail trail immediately, with log file integrity validation enabled and logs delivered to a dedicated, access-controlled S3 bucket. Consider integrating with CloudWatch Logs for real-time alerting.

2. IAM Findings

🟠 IAM-001 — Overly Permissive Policy: ai_scanner_user

Field Detail
Service IAM
Severity HIGH
Resource ai_scanner_user
Resource Type IAM User

Description: The IAM user ai_scanner_user has been assigned a managed policy that grants excessively broad permissions. Programmatic or interactive users should operate under the principle of least privilege, with permissions scoped to only the specific actions and resources required for their function.

Impact: - Potential for privilege abuse if credentials are compromised or misused - Expanded blast radius in the event of account compromise - Violates the AWS Well-Architected Framework security pillar

5. Remediation Recommendations

Priority 1 — Immediate Actions (Within 24 Hours)

✅ Enable AWS CloudTrail

1. Navigate to AWS Console → CloudTrail → Create Trail 2. Enable for all regions (multi-region trail) 3. Enable log file integrity validation 4. Deliver logs to a dedicated S3 bucket with restrictive bucket policy 5. Enable CloudWatch Logs integration for real-time alerting 6. Consider enabling CloudTrail Insights for anomaly detection

✅ Restrict launch-wizard-6 (All Traffic Open)

1. Navigate to EC2 → Security Groups → sg-003020051a81a8c82 2. Remove the inbound rule permitting all traffic from 0.0.0.0/0 3. Add only the specific ports and source IPs required for operation 4. If unused, delete the security group entirely

✅ Restrict ELK Stack Ports (9200, 5601, 5044)

``` Affected groups: sg-0b1c35cdbed3ecd32, sg-0a0df35eef090ae38, sg-0ea3e4a05a67ca02a, sg-086216d42764aa08e

  1. Remove public inbound rules for ports 9200, 5601, and 5044
  2. Restrict access to specific trusted IP ranges or VPN CIDR blocks
  3. Place Elasticsearch behind a VPC with no direct internet access
  4. Use an application load balancer with authentication for Kibana ```

Priority 2 — Short-Term Actions (Within 72 Hours)

✅ Restrict SSH (Port 22) and RDP (Port 3389)

``` Affected groups: sg-0b2da4349ee7bc901 (SSH), sg-0ddc1c39a3fcc2123 (SSH), sg-0c3704fd44e21ac07 (RDP)

  1. Remove inbound rules for ports 22 and 3389 from 0.0.0.0/0
  2. Restrict to specific corporate IP ranges or VPN CIDR blocks
  3. Consider migrating SSH/RDP access to AWS Systems Manager Session Manager (eliminates need for open SSH/RDP entirely)
  4. Enable MFA for any remaining direct access ```

✅ Apply Least Privilege to ai_scanner_user and terraform-bootstrap

1. Review current attached policies for both principals 2. Use AWS IAM Access Analyzer to identify unused permissions 3. Replace broad managed policies with custom policies scoped to: - Specific actions required for the use case - Specific resource ARNs where possible 4. For terraform-bootstrap, scope to only the AWS services Terraform manages 5. Enable periodic access reviews using IAM Access Analyzer

✅ Scope EC2InstanceConnectRole Permissions

1. Review attached managed policies 2. Remove the broad SSM/S3 managed policy 3. Create a custom policy with only required permissions: - ec2-instance-connect:SendSSHPublicKey on specific instance ARNs - ssm:StartSession scoped to specific instance ARNs 4. Restrict S3 permissions to specific buckets if required 5. Set resource conditions to limit to specific regions and account IDs

Priority 3 — Medium-Term Actions (Within 2 Weeks)

✅ Review AWS Service-Linked Roles

``` For: AWSServiceRoleForResourceExplorer, AWSServiceRoleForSupport, AWSServiceRoleForTrustedAdvisor

  1. Confirm each service is actively in use and required
  2. If AWS Support is not required at the current tier, downgrade or disable
  3. If Resource Explorer is not in active use, consider disabling the service
  4. Document justification for each service-linked role in use
  5. Note: AWS-managed service role policies cannot be modified directly; control is exercised by enabling/disabling the underlying service ```

✅ Audit and Rename Launch Wizard Security Groups

1. "launch-wizard-*" names indicate auto-generated groups from EC2 launch wizard 2. These are often created ad-hoc and not maintained 3. Conduct a full audit: identify which instances use each group 4. Replace with environment-specific, purpose-named security groups 5. Implement a security group naming convention and tagging policy 6. Remove unused security groups

✅ Implement Preventive Controls

1. Enable AWS Config with managed rules: - restricted-ssh - restricted-common-ports - iam-no-inline-policy-check - cloudtrail-enabled 2. Deploy AWS Security Hub with CIS AWS Foundations Benchmark 3. Implement AWS Organizations Service Control Policies (SCPs) to prevent: - CloudTrail deletion/disabling - Public S3 bucket creation - Security group rules permitting 0.0.0.0/0 on sensitive ports
