FINDING-001 — No CloudTrail Trails Found

Field Detail
Severity 🔴 CRITICAL
Service CloudTrail
Resource N/A (Account-Wide)

Description

No AWS CloudTrail trails were detected in this environment. CloudTrail is the primary audit and logging service for AWS API activity. Without it, there is no record of who performed what actions, when, or from where within the AWS account.

Risk

  • Complete lack of audit trail for all AWS API calls
  • Inability to detect unauthorized access, insider threats, or suspicious activity
  • Non-compliance with virtually all major security frameworks (SOC 2, PCI-DSS, HIPAA, NIST, ISO 27001)
  • Incident response and forensic investigations are severely impaired or impossible
  • Regulatory and legal exposure in the event of a breach

Remediation

  1. Enable CloudTrail immediately in all AWS regions with a multi-region trail.
  2. Configure the trail to deliver logs to a dedicated, versioning-enabled, MFA-delete-protected S3 bucket.
  3. Enable CloudTrail log file integrity validation to detect tampering.
  4. Optionally forward logs to CloudWatch Logs for real-time alerting.
  5. Enable AWS Config in parallel to capture resource configuration change history.

```bash

Example: Create a multi-region CloudTrail trail

aws cloudtrail create-trail \ --name org-wide-audit-trail \ --s3-bucket-name \ --is-multi-region-trail \ --enable-log-file-validation

aws cloudtrail start-logging --name org-wide-audit-trail ```

FINDING-002 — IAM User ai_scanner_user Has Overly Permissive Managed Policy

Field Detail
Severity 🟠 HIGH
Service IAM
Resource ai_scanner_user

Description

The IAM user ai_scanner_user has been granted an overly permissive managed policy. Granting excessive permissions to IAM users violates the principle of least privilege and significantly increases the blast radius of a credential compromise.

Risk

  • Compromised credentials could allow an attacker broad access to AWS resources
  • Permissions likely exceed what is required for the user's intended function
  • IAM users with long-lived access keys are higher-risk than role-based access

Remediation

  1. Audit the policies attached to ai_scanner_user using AWS IAM Access Analyzer.
  2. Identify the minimum set of actions and resources required for the scanner's function.
  3. Replace broad managed policies with a custom least-privilege policy.
  4. If the user is used programmatically, consider replacing it with an IAM Role assumed via EC2, Lambda, or OIDC federation.
  5. Rotate any existing access keys and enforce MFA if the user remains in use.

json // Example: Restrictive policy scoped to specific actions and resources { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": [ "ec2:Describe*", "iam:List*", "iam:Get*" ], "Resource": "*" } ] }

FINDING-003 — IAM Role terraform-bootstrap Has Overly Permissive Managed Policy

Field Detail
Severity 🟠 HIGH
Service IAM
Resource terraform-bootstrap

Description

The terraform-bootstrap IAM role or user has been assigned an overly permissive managed policy. Terraform bootstrap roles frequently carry AdministratorAccess or similarly broad permissions, which is appropriate only during initial setup and should be locked down post-provisioning.

Risk

  • If CI/CD pipelines or automation use this role, a pipeline compromise could result in full account takeover
  • Broad permissions may persist well beyond the bootstrapping phase
  • Violates least-privilege and infrastructure security best practices

Remediation

  1. Determine whether the bootstrap role is still actively needed. If not, delete or disable it.
  2. If still required, scope permissions to only the specific resources and actions needed by Terraform (e.g., specific S3 buckets, DynamoDB tables for state locking, and scoped deployment permissions).
  3. Use permission boundaries to cap the maximum effective permissions.
  4. Restrict who or what can assume this role using a tightly scoped trust policy.

FINDING-004 — AWSServiceRoleForResourceExplorer Has Overly Permissive Policy

Field Detail
Severity 🟠 HIGH
Service IAM
Resource AWSServiceRoleForResourceExplorer

Description

The AWS Service-Linked Role for Resource Explorer has been identified as having overly permissive policies, including: - 12 actions across organizations and resource-explorer-2 on all resources - 725 actions across 188+ services (including access-analyzer, acm, acm-pca, airflow, amplify, and more) on all resources - 139 actions across 44+ services (including backup, bedrock, budgets, cleanrooms, and more) on all resources

Risk

  • The scope of read/list permissions is extremely broad, covering the majority of AWS services
  • If the Resource Explorer service or its configuration is misconfigured or abused, sensitive resource metadata could be exposed
  • The broad Resource: * scope is a concern in multi-account or sensitive environments

Remediation

Note: AWS Service-Linked Roles (SLRs) are managed by AWS and their policies cannot be directly modified. However, the following steps can reduce exposure:

  1. Evaluate necessity: Determine if AWS Resource Explorer is actively required. If not, disable the service — this will remove the associated SLR.
  2. Apply SCPs: Use AWS Organizations Service Control Policies (SCPs) to limit what Resource Explorer can access at the organizational level.
  3. Review Resource Explorer indexing scope: Limit indexing to specific regions and resource types via Resource Explorer configuration settings.
  4. Monitor SLR activity: Use CloudTrail (once enabled — see FINDING-001) to audit actions taken by this role.

FINDING-007 — EC2InstanceConnectRole Has Overly Permissive Policies

Field Detail
Severity 🟠 HIGH
Service IAM
Resource EC2InstanceConnectRole

Description

The EC2InstanceConnectRole has been flagged across two overlapping policy findings with broad permissions including: - ec2-instance-connect: 1 action on all resources - ssm: 14 actions on all resources - ssmmessages: 4 actions on all resources - ec2messages: 6 actions on all resources - cloudwatch: 1 action on all resources - ec2: 1 action on all resources - ds (Directory Service): 2 actions on all resources - logs: 5 actions on all resources - s3: 8 actions on all resources

The most significant concern is the use of Resource: * across all of these permissions, meaning the role can act on any resource in the account, not just the specific instances it is intended to serve.

Risk

  • An EC2 instance using this role could access SSM parameters, S3 buckets, and CloudWatch logs beyond its intended scope
  • Instance compromise could allow lateral movement via SSM to other EC2 instances
  • S3 access on all resources is particularly dangerous if sensitive data buckets exist

Remediation

  1. Scope S3 permissions to only the specific bucket(s) required (e.g., for SSM Agent logs).
  2. Scope SSM permissions to specific instance ARNs where possible.
  3. Scope CloudWatch Logs permissions to the specific log group(s) used by the instance.
  4. Replace the broad "Resource": "*" with specific ARNs for each permission type.

json // Example: Scoped S3 access for SSM Agent logs { "Effect": "Allow", "Action": ["s3:PutObject", "s3:GetObject"], "Resource": "arn:aws:s3:::my-ssm-logs-bucket/instance-logs/*" }

FINDING-008 — Security Group EC2-Video-Edits Allows Public RDP Access (Port 3389)

Field Detail
Severity 🟠 HIGH
Service EC2
Resource EC2-Video-Edits (sg-08134cc1e2bf22109)

Description

The security group EC2-Video-Edits has an inbound rule permitting unrestricted TCP access on port 3389 (RDP — Remote Desktop Protocol) from the internet (0.0.0.0/0 or ::/0). This exposes any EC2 instances associated with this security group to brute-force attacks, credential stuffing, and exploitation of RDP vulnerabilities from any source on the internet.

Risk

  • RDP exposed to the internet is one of the most commonly exploited attack vectors in cloud environments
  • Susceptible to brute-force attacks, BlueKeep-style vulnerabilities, and unauthorized access
  • A successful compromise grants full GUI-level control of the Windows instance
  • Associated instances may serve as a pivot point for lateral movement into the broader environment

Remediation

  1. Immediately remove the 0.0.0.0/0 inbound rule on port 3389.
  2. Restrict RDP access to known, static IP ranges (e.g., corporate office IPs or VPN egress IPs only).
  3. Prefer AWS Systems Manager Session Manager over RDP for instance access — this eliminates the need for open inbound ports entirely.
  4. If RDP must remain accessible, place instances behind a bastion host or VPN, and require MFA for VPN access.
  5. Enable AWS GuardDuty to detect brute-force RDP attempts.

```bash

Revoke the overly permissive RDP rule

aws ec2 revoke-security-group-ingress \ --group-id sg-08134cc1e2bf22109 \ --protocol tcp \ --port 3389 \ --cidr 0.0.0.0/0

Add a restrictive rule for a specific IP range (replace with your IP)

aws ec2 authorize-security-group-ingress \ --group-id sg-08134cc1e2bf22109 \ --protocol tcp \ --port 3389 \ --cidr /32 ```

General Recommendations

  2. Enable Amazon GuardDuty — Provides continuous threat detection across CloudTrail, VPC Flow Logs, and DNS logs.
  3. Implement AWS Organizations SCPs — Apply guardrails at the organizational level to enforce least privilege regardless of individual account configurations.
  4. Adopt IAM Access Analyzer — Continuously analyze policies to identify external and cross-account access risks.
