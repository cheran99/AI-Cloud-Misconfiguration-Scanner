🔴 FINDING-001 — No CloudTrail Trails Found

Field Detail
Service CloudTrail
Severity CRITICAL
Resource N/A (Account-Wide)

Description

No AWS CloudTrail trails are configured in this account. CloudTrail is the foundational auditing and logging service for AWS, recording all API calls made within the account — including who made them, from where, and when.

Risk

Without CloudTrail enabled, the organization has no visibility into user activity, resource changes, or security events. This means: - Insider threats and unauthorized access cannot be detected or investigated. - Compliance requirements (e.g., PCI-DSS, SOC 2, HIPAA, ISO 27001) are almost certainly being violated. - Incident response and forensic investigations are severely impaired or impossible. - Malicious actors can operate within the account without leaving an auditable trace.

Recommendation

  1. Immediately enable a CloudTrail trail covering all regions.
  2. Configure the trail to log both management events and data events for sensitive resources (e.g., S3, Lambda).
  3. Store logs in a dedicated, access-controlled S3 bucket with server-side encryption (SSE-KMS) enabled.
  4. Enable log file validation to detect tampering.
  5. Consider integrating CloudTrail with CloudWatch Logs or a SIEM for real-time alerting.
  6. Apply an S3 bucket policy that prevents deletion of log files and restricts access to authorized principals only.

🟠 FINDING-002 — ai_scanner_user Has Overly Permissive IAM Policy

Field Detail
Service IAM
Severity HIGH
Resource ai_scanner_user

Description

The IAM user ai_scanner_user has been assigned a managed policy that grants permissions exceeding what is necessary for its intended function.

Risk

Overly permissive IAM policies violate the principle of least privilege. If this user's credentials are compromised (e.g., via credential leakage, phishing, or exposed access keys), an attacker could leverage the excessive permissions to escalate privileges, exfiltrate data, or cause widespread damage.

Recommendation

  1. Conduct an IAM Access Analyzer review or use the IAM Policy Simulator to determine what permissions are actually used.
  2. Replace the broad managed policy with a custom, least-privilege policy scoped to only the actions and resources required.
  3. Review and rotate any access keys associated with this user.
  4. Consider replacing long-term credentials with IAM Roles and temporary credentials where possible.
  5. Enable MFA for this user if it is used for console access.

Field Detail
Service IAM
Severity HIGH
Resource terraform-bootstrap

Description

The IAM entity terraform-bootstrap has been assigned a managed policy that grants permissions beyond what is required for infrastructure bootstrapping operations.

Risk

Terraform bootstrap roles/users are particularly high-value targets because they are frequently used to provision infrastructure and may be used in CI/CD pipelines. Overly broad permissions associated with this entity could allow: - Unauthorized infrastructure creation or destruction. - Privilege escalation through the creation of new IAM users or roles. - Exposure of sensitive state or configuration data.

Recommendation

  1. Scope the policy to only the AWS services and actions needed during the Terraform bootstrapping phase.
  2. If used in a CI/CD pipeline, ensure the credentials are stored securely (e.g., in AWS Secrets Manager or as environment-specific secrets in the pipeline tool).
  3. Implement permission boundaries to cap the maximum permissions this entity can grant to other principals.
  4. Audit usage via CloudTrail once enabled (see FINDING-001) to identify any misuse.
  5. Consider using short-lived credentials via OIDC federation instead of static access keys for CI/CD workflows.

Description

  3. Monitor the role's activity via CloudTrail for any anomalous behavior.
  4. Raise a concern with AWS Support regarding the scope of permissions granted to this service-linked role and request guidance on tightening the policy.

🟠 FINDING-007 — EC2InstanceConnectRole Has Overly Permissive Policies

Field Detail
Service IAM
Severity HIGH
Resource EC2InstanceConnectRole

Description

The EC2InstanceConnectRole has been assigned multiple overly permissive managed policies granting permissions on all resources across numerous services, including: - ec2-instance-connect (1 action) - ssm (14 actions), ssmmessages (4 actions), ec2messages (6 actions) - cloudwatch (1 action), ec2 (1 action), ds (2 actions) - logs (5 actions), s3 (8 actions)

Risk

While some of these permissions are expected for Systems Manager Session Manager functionality, the "all resources" scope on services like S3 (8 actions) and EC2 is unnecessarily broad. If this role is attached to an EC2 instance and that instance is compromised, an attacker could leverage these permissions to: - Read or exfiltrate data from any S3 bucket in the account. - Query or manipulate EC2 resources account-wide. - Access SSM parameter store values that may contain secrets.

Recommendation

  1. Scope S3 permissions to only the specific buckets the instance needs to access.
  2. Scope EC2 permissions to only the necessary instance(s) or resource tags.
  3. Use resource-level conditions (e.g., aws:ResourceTag) to limit the blast radius.
  4. Review whether all listed actions are actually required and remove unnecessary permissions.
  5. Consider using VPC Endpoints for SSM and S3 to avoid traffic traversing the public internet.

🟠 FINDING-008 — Security Group EC2-Video-Edits Exposes RDP to the Internet

Field Detail
Service EC2
Severity HIGH
Resource EC2-Video-Edits (sg-08134cc1e2bf22109)
Protocol TCP
Port 3389 (RDP)
Source 0.0.0.0/0 (All Internet)

Description

The security group EC2-Video-Edits allows unrestricted inbound access on TCP port 3389 (Remote Desktop Protocol) from the entire internet (0.0.0.0/0).

Risk

RDP is one of the most commonly exploited protocols on the internet. Exposing port 3389 publicly creates a high-risk attack vector for: - Brute-force and credential stuffing attacks targeting Windows login credentials. - Exploitation of RDP vulnerabilities such as BlueKeep (CVE-2019-0708) and related CVEs. - Ransomware delivery, as RDP is the #1 initial access vector for ransomware attacks. - Lateral movement if the instance is compromised and connected to other internal resources.

Recommendation

  1. Immediately remove the 0.0.0.0/0 inbound rule for port 3389.
  2. If RDP access is genuinely required:
  3. Restrict access to specific, known IP addresses (e.g., corporate office IP ranges or VPN egress IPs).
  4. Alternatively, require users to connect via a VPN or AWS Client VPN before accessing the instance, removing any need for public RDP exposure.
  5. Strongly consider replacing direct RDP with AWS Systems Manager Session Manager, which provides secure, audited shell/RDP access without opening any inbound ports.
  6. Review what other ports are open on this security group and apply least-privilege network access controls.
  7. Enable Amazon GuardDuty to detect any brute-force or unauthorized access attempts that may have already occurred.
