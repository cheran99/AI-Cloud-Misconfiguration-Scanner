Critical Finding

⛔ CF-001 — No AWS CloudTrail Trails Found

Field Details
Service CloudTrail
Severity CRITICAL
Resource N/A (Account-wide)

Description

No AWS CloudTrail trails are configured in this AWS account. CloudTrail is the primary mechanism for recording API activity and governance events across all AWS services. Without it, there is zero visibility into who is performing actions in the environment, making incident detection, forensic investigation, and compliance reporting impossible.

Risk Impact

  • No audit log of API calls, console logins, or resource changes.
  • Inability to detect unauthorized access, privilege escalation, or data exfiltration.
  • Non-compliance with regulatory frameworks including PCI-DSS, HIPAA, SOC 2, ISO 27001, and CIS AWS Foundations Benchmark.
  • Significant hindrance to any incident response effort.

Recommended Remediation

  1. Immediately create a multi-region CloudTrail trail to capture all API activity across all AWS regions.
  2. Enable log file integrity validation to detect tampering.
  3. Store CloudTrail logs in a dedicated, locked-down S3 bucket with versioning and MFA Delete enabled.
  4. Enable CloudWatch Logs integration to support real-time alerting on sensitive API calls.
  5. Consider enabling CloudTrail Insights to detect unusual API activity automatically.

Example Terraform Configuration

High Severity Findings

🟠 IAM-001 — ai_scanner_user Has Overly Permissive Managed Policy

Field Details
Service IAM
Severity HIGH
Resource ai_scanner_user (IAM User)

Description

The IAM user ai_scanner_user has been granted an overly permissive managed policy. This violates the Principle of Least Privilege (PoLP), granting more permissions than are necessary for the user's intended function.

Risk Impact

  • If credentials are compromised, an attacker gains broad access to AWS resources.
  • Increases the blast radius of any accidental or malicious action performed by this user.
  • IAM users with static, long-lived credentials represent a higher risk surface than roles.

Recommended Remediation

  1. Audit the exact permissions granted to ai_scanner_user using IAM Access Analyzer or the Policy Simulator.
  2. Replace the overly permissive managed policy with a custom, least-privilege policy scoped to only the actions and resources required for the scanning function.
  3. Evaluate whether this workload can use an IAM Role instead of a long-lived IAM user.
  4. Enable IAM Access Advisor to identify and remove unused permissions.
  5. Rotate and audit any access keys associated with this user.

🟠 IAM-002 — terraform-bootstrap Has Overly Permissive Managed Policy

Field Details
Service IAM
Severity HIGH
Resource terraform-bootstrap (IAM User/Role)

Description

The terraform-bootstrap principal has been assigned an overly permissive managed policy. Terraform bootstrap identities are commonly granted broad permissions to facilitate initial infrastructure provisioning, but these permissions are frequently not reviewed or tightened after initial setup.

Risk Impact

  • If the Terraform credentials are exposed (e.g., in CI/CD pipelines, state files, or version control), an attacker gains wide-ranging access.
  • Overly broad Terraform identities can be used to create backdoor users, roles, or resources.
  • Violates the principle of least privilege and infrastructure security best practices.

Recommended Remediation

  1. Define and enforce a scoped policy that grants only the AWS actions required for the specific Terraform modules in use.
  2. Use IAM permission boundaries to limit the maximum permissions the Terraform identity can delegate.
  3. Replace static IAM user credentials with short-lived role credentials via OIDC federation (e.g., GitHub Actions OIDC, AWS IAM Roles Anywhere).
  4. Regularly review Terraform state files to ensure sensitive data is encrypted and access is restricted.
  5. Use Service Control Policies (SCPs) at the AWS Organizations level to constrain this identity's permissions.

🟠 IAM-006 & IAM-007 — EC2InstanceConnectRole Has Overly Permissive Managed Policies

Field Details
Service IAM
Severity HIGH
Resource EC2InstanceConnectRole (IAM Role)

Description

The EC2InstanceConnectRole has two separate overly permissive policy findings:

Finding 1 — EC2 Instance Connect Access: | Actions | Service | |---|---| | 1 action | ec2-instance-connect — scoped to all resources |

Finding 2 — Systems Manager & Supporting Services: | Actions | Service | |---|---| | 14 actions | ssm | | 8 actions | s3 | | 6 actions | ec2messages | | 5 actions | logs | | 4 actions | ssmmessages | | 2 actions | ds | | 1 action | cloudwatch | | 1 action | ec2 |

All permissions are scoped to all resources (*), meaning the role can connect to and manage any EC2 instance in the account, not just specific ones.

Risk Impact

  • If this role is assumed by an unauthorized user or compromised entity, it provides unrestricted SSH/RDP access to all EC2 instances.
  • Broad SSM permissions (ssm:* across all resources) can enable command execution on any instance, configuration changes, and session recording manipulation.
  • Unrestricted S3 access from this role could expose or exfiltrate data stored in SSM-related buckets.

Recommended Remediation

  1. Restrict resource scope: Update both policies to target specific EC2 instance ARNs, SSM parameter paths, and S3 bucket ARNs rather than using wildcard (*) resources.

json // Example: Restrict EC2 Instance Connect to specific instances { "Effect": "Allow", "Action": "ec2-instance-connect:SendSSHPublicKey", "Resource": "arn:aws:ec2:REGION:ACCOUNT_ID:instance/i-xxxxxxxxxxxxxxxxx", "Condition": { "StringEquals": { "ec2:osuser": "ec2-user" } } }

  1. Apply IAM Conditions to restrict instance connect to specific instance IDs, OS users, or tags (e.g., ec2:ResourceTag/Environment: production).
  2. Restrict SSM actions to only those required (e.g., ssm:StartSession, ssm:TerminateSession) and scope to specific instance resource ARNs.
  3. Enable SSM Session Manager logging to CloudWatch and S3 to maintain an audit trail of all sessions.
  4. Consider implementing AWS IAM Identity Center (SSO) with time-limited access for instance access rather than a persistent broad role.
