Detailed Findings

Finding 1 — IAM: Overly Permissive Policy on ai_scanner_user

Field Detail
Resource ai_scanner_user
Service IAM
Severity 🔴 HIGH

Description

The IAM user ai_scanner_user has been granted a managed policy that is considered overly permissive. Granting excessive permissions to IAM users violates the Principle of Least Privilege (PoLP) and increases the blast radius in the event of credential compromise.

Risk

If the credentials associated with ai_scanner_user are leaked, stolen, or misused, an attacker could leverage the broad permissions to access, modify, or exfiltrate sensitive resources across the AWS environment.

Recommendations

  • Audit the current policies attached to ai_scanner_user and identify all granted permissions.
  • Remove the overly permissive managed policy and replace it with a custom, scoped-down policy that grants only the actions required for the scanner's specific function.
  • Enable MFA for the user if it is not already enforced.
  • Rotate access keys and review CloudTrail logs for any suspicious activity associated with this user.
  • Consider converting to an IAM Role-based approach instead of a long-lived IAM user, where applicable.

Finding 2 — IAM: Overly Permissive Policy on terraform-bootstrap

Field Detail
Resource terraform-bootstrap
Service IAM
Severity 🔴 HIGH

Description

The IAM identity terraform-bootstrap has been assigned a managed policy that is overly permissive. Bootstrap identities used for infrastructure provisioning are high-value targets because they commonly require broad access during initial setup but are rarely restricted after the bootstrapping phase is complete.

Risk

Terraform bootstrap roles or users often carry near-administrator-level permissions. If not properly scoped post-deployment, they represent a persistent attack surface. Compromise of this identity could allow an attacker to reshape the entire cloud infrastructure.

Recommendations

  • Review and audit all permissions currently attached to terraform-bootstrap.
  • Determine if the bootstrapping phase is complete; if so, significantly reduce permissions to a minimal operational set or disable the identity entirely.
  • If ongoing Terraform automation is required, migrate to an IAM Role with an appropriate trust policy and session conditions.
  • Implement IAM permission boundaries to cap the maximum permissions this identity can exercise.
  • Ensure all Terraform state files are stored securely with appropriate access controls.

Finding 6 — IAM: Overly Permissive EC2 Instance Connect Policy (EC2 Action)

Field Detail
Resource EC2InstanceConnectRole
Service IAM
Severity 🔴 HIGH

Description

The EC2InstanceConnectRole has a policy granting 1 action for the ec2-instance-connect service scoped to all resources (*). While a single action may seem minimal, allowing ec2-instance-connect permissions against all EC2 resources (rather than specific instance ARNs) means any entity assuming this role can connect to any EC2 instance in the account.

Risk

Unrestricted EC2 Instance Connect access across all instances eliminates instance-level access control. An attacker or compromised process assuming this role could gain SSH access to any EC2 instance in the account regardless of its sensitivity.

Recommendations

  • Restrict the ec2-instance-connect:SendSSHPublicKey (or equivalent) action to specific instance ARNs or resource tags rather than Resource: *.
  • Use condition keys such as ec2:ResourceTag to limit connections to instances with specific authorized tags.
  • Enforce instance-level policies and security groups independently of IAM to provide defense in depth.

Finding 8 — S3: Bucket Encryption Not Using AWS KMS

Field Detail
Resource aws-cloudtrail-logs-211125321299-2a1a565d
Service S3
Severity 🔴 HIGH

Description

The S3 bucket aws-cloudtrail-logs-211125321299-2a1a565d, which is used to store CloudTrail audit logs, is not encrypted using AWS KMS (SSE-KMS). The bucket may be using SSE-S3 (AES-256) or may have no encryption policy enforced. For an audit log repository, the absence of KMS encryption represents a significant security and compliance gap.

Risk

  • Compliance Violations: Regulatory frameworks (e.g., PCI DSS, HIPAA, SOC 2, ISO 27001) commonly require that audit logs be encrypted with customer-managed keys (CMKs) to ensure key access can be independently controlled and audited.
  • Unauthorized Access to Audit Logs: Without KMS-based encryption, there is no CMK-level access control layer. Anyone with S3 GetObject permissions can read log data without needing KMS kms:Decrypt authorization.
  • Key Auditability: SSE-KMS provides a separate, auditable key usage trail in AWS CloudTrail (KMS API calls), which SSE-S3 does not provide.

Recommendations

  • Create or identify an existing AWS KMS Customer Managed Key (CMK) appropriate for encrypting CloudTrail logs.
  • Update the S3 bucket's default encryption settings to use SSE-KMS with the designated CMK.
  • Update the CloudTrail trail configuration to use the same KMS key for log file encryption.
  • Apply a bucket policy that denies s3:PutObject requests that do not specify aws:kms as the server-side encryption method.
  • Restrict access to the KMS CMK via key policy to only authorized principals (e.g., CloudTrail service, security team).
  • Enable KMS key rotation for the CMK.

Example S3 Bucket Policy Condition (enforcement): json { "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" } } }

Finding 9 — CloudTrail: Trail Not Logging (Video-Edits-Trail)

Field Detail
Resource Video-Edits-Trail
Service CloudTrail
Severity 🔴 HIGH

Description

The CloudTrail trail Video-Edits-Trail has been identified as not actively logging. This means that API activity and events within the scope of this trail are not being captured, creating a blind spot in the environment's audit and incident response capabilities.

Risk

  • Loss of Audit Visibility: All AWS API calls, management events, and data events within the trail's scope are going unrecorded. This makes it impossible to reconstruct what actions were taken, by whom, and when.
  • Incident Response Impairment: In the event of a security incident, the absence of logs for this trail severely hampers forensic investigation.
  • Compliance Violations: Most regulatory frameworks require continuous and complete audit logging. A disabled trail is likely a direct compliance violation.
  • Attacker Opportunity: Sophisticated attackers may intentionally disable CloudTrail trails to cover their tracks. The current inactive state of this trail should be investigated to rule out malicious disablement.

Recommendations

  • Immediately re-enable the Video-Edits-Trail CloudTrail trail via the AWS Console or CLI: bash aws cloudtrail start-logging --name Video-Edits-Trail
  • Investigate the root cause of why this trail is not logging:
  • Was it manually disabled? If so, by whom and when? (Check CloudTrail management events in other active trails.)
  • Was it disabled via automation or IaC changes?
  • Configure a CloudWatch Alarm or EventBridge Rule to alert immediately if any CloudTrail trail is stopped or deleted in the future.
  • Enable CloudTrail log file validation to detect tampering with existing log files.
  • Ensure this trail's logs are being delivered to the KMS-encrypted S3 bucket (see Finding 8) once re-enabled.
  • Consider enabling CloudTrail Insights on this trail to detect unusual API activity automatically.

Preventive Control — Example EventBridge Rule: Monitor for StopLogging events and trigger an SNS alert or automated remediation Lambda function.
