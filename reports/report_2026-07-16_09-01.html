Detailed Findings

FINDING-001 — HIGH

Resource: ai_scanner_user Service: IAM Issue: The IAM user ai_scanner_user has been assigned an overly permissive managed policy.

Risk: Granting excessive permissions to an IAM user — particularly one associated with an automated scanner — creates a significant risk of privilege abuse. If the user's credentials are compromised or the scanner is misconfigured, an attacker could leverage these permissions to access, modify, or exfiltrate sensitive resources across the AWS environment.

Recommendation: - Audit the managed policies attached to ai_scanner_user and identify all granted actions. - Replace broad managed policies with a custom, least-privilege policy scoped to only the actions required for scanning operations (e.g., read-only Describe*, List*, Get* permissions on specific services). - Consider using IAM roles with temporary credentials instead of long-lived IAM user credentials. - Enable IAM Access Analyzer to continuously monitor for overly permissive policies.

FINDING-002 — HIGH

Resource: terraform-bootstrap Service: IAM Issue: The IAM entity terraform-bootstrap has been assigned an overly permissive managed policy.

Risk: Terraform bootstrap roles or users are commonly granted broad administrative permissions during initial infrastructure provisioning. Leaving such permissions in place post-bootstrap significantly increases the blast radius of a compromise, potentially allowing full infrastructure takeover.

Recommendation: - Review the current policy attached to terraform-bootstrap and determine whether broad permissions are still operationally necessary. - Scope permissions down to only the specific services and actions required for ongoing Terraform operations. - If the bootstrap phase is complete, consider disabling or deleting the entity entirely. - If the entity must persist, enforce MFA and restrict usage to specific IP ranges or conditions using IAM policy conditions. - Rotate any associated access keys immediately if they have not been rotated recently.

FINDING-007 — HIGH

Resource: aws-cloudtrail-logs-211125321299-2a1a565d Service: S3 Issue: The S3 bucket used to store AWS CloudTrail logs is not encrypted using AWS KMS. The bucket either uses SSE-S3 (AES-256) or has no encryption configured.

Risk: CloudTrail logs are among the most sensitive artifacts in an AWS environment — they record every API call made across the account and are frequently used for security investigations and compliance audits. Without KMS encryption: - There is no ability to enforce cryptographic access controls using KMS key policies. - Logs cannot be protected with customer-managed key rotation and auditing. - Compliance frameworks (e.g., PCI-DSS, HIPAA, SOC 2) may require KMS-based encryption for audit logs. - If the bucket is inadvertently made public or accessed by an unauthorized principal, log data is immediately readable.

Recommendation: - Enable SSE-KMS encryption on the aws-cloudtrail-logs-211125321299-2a1a565d bucket using a Customer Managed Key (CMK). - Create a dedicated KMS key for CloudTrail log encryption with a strict key policy allowing access only to CloudTrail, authorized administrators, and security/audit roles. - Enable KMS key rotation (annually at minimum). - Update the CloudTrail trail configuration to use the KMS key for log file encryption at the trail level as well (via the KMSKeyId setting in CloudTrail). - Ensure the S3 bucket policy denies s3:PutObject requests that do not include the required KMS encryption header.

Example bucket policy condition to enforce KMS encryption: json { "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" } } }

Strategic Recommendations

1. Enforce Least Privilege Across All IAM Entities

Conduct a full IAM audit using AWS IAM Access Analyzer and AWS IAM Access Advisor. Identify and remove unused permissions, replace wildcard resource ARNs (*) with specific resource ARNs, and establish a policy review cadence (quarterly minimum).

2. Implement AWS Service Control Policies (SCPs)

Use SCPs at the AWS Organizations level to enforce guardrails on service-linked roles and restrict which accounts can enable services like Resource Explorer, Trusted Advisor, and AWS Support tiers.

3. Protect CloudTrail Integrity

Beyond KMS encryption, ensure CloudTrail log file validation is enabled, the log bucket is not publicly accessible, object-level logging is enabled on the CloudTrail bucket itself, and access to the bucket is logged and alerted on.

4. Adopt Role-Based Access with Temporary Credentials

Replace long-lived IAM user credentials (ai_scanner_user, terraform-bootstrap) with IAM roles and temporary credentials (AWS STS). Use instance profiles, ECS task roles, or federated identity where applicable.

5. Establish Continuous Monitoring

Deploy AWS Security Hub with the AWS Foundational Security Best Practices standard enabled to receive continuous, automated findings similar to those in this report. Integrate findings with your SIEM or alerting platform for real-time response.

Conclusion

All 8 findings identified in this assessment are classified as HIGH severity and warrant prompt attention. The predominance of overly permissive IAM policies — spanning both user-created entities and AWS service-linked roles — reflects a systemic need to implement and enforce a least-privilege IAM strategy. The unencrypted CloudTrail log bucket compounds the risk by potentially exposing audit trail data that could aid attackers in covering their tracks.

Addressing the P1 findings (ai_scanner_user, terraform-bootstrap, and the CloudTrail bucket) can be accomplished with relatively low effort and should be prioritized immediately. Service-linked role findings, while requiring coordination with AWS service configurations and SCPs, should be addressed as part of a near-term security hardening sprint.
