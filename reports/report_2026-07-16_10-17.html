Detailed Findings

Finding 1 — ai_scanner_user

Field Detail
Resource ai_scanner_user
Resource Type IAM User
Severity 🔴 HIGH
Service IAM

Description

The IAM user ai_scanner_user has been assigned a managed policy that grants excessively broad permissions. This violates the principle of least privilege and exposes the account to potential misuse if the user's credentials are compromised.

Risk

  • Unauthorized access to sensitive AWS resources via compromised credentials.
  • Potential for lateral movement or privilege escalation within the account.

Recommendation

  • Audit the managed policies attached to ai_scanner_user and restrict permissions to only those required for its intended function.
  • Consider replacing broad managed policies with a tightly scoped custom inline policy.
  • Enforce multi-factor authentication (MFA) on the user.
  • Rotate access keys regularly and consider replacing long-lived credentials with IAM Roles where possible.

Finding 2 — terraform-bootstrap

Field Detail
Resource terraform-bootstrap
Resource Type IAM User / Role
Severity 🔴 HIGH
Service IAM

Description

The terraform-bootstrap resource has been assigned a managed policy with overly permissive access. Infrastructure automation roles are high-value targets and should be scoped tightly to only the permissions required during bootstrapping operations.

Risk

  • Compromise of Terraform automation credentials could grant an attacker broad control over the AWS environment.
  • Persistent over-permissioned automation identities increase the blast radius of any security incident.

Recommendation

  • Limit the permissions of terraform-bootstrap to only those actions required during the Terraform bootstrapping lifecycle.
  • Once bootstrapping is complete, consider deactivating or deleting this identity.
  • Use IAM Permission Boundaries to cap the maximum permissions this identity can exercise.
  • Store and manage credentials in a secrets manager (e.g., AWS Secrets Manager or HashiCorp Vault).

Risk

  • This role represents the largest attack surface of any identity identified in this assessment.
  • AWS Support roles, by design, require broad visibility; however, the scope should be carefully validated to ensure no write or destructive actions are included.

Recommendation

  • Conduct a detailed review of all policy documents attached to AWSServiceRoleForSupport to distinguish read-only from write/destructive actions.
  • Verify that this role aligns with AWS's documented Support service-linked role policies.
  • Enable AWS CloudTrail logging and set up alerting for any unexpected API calls made by this role.
  • If AWS Business or Enterprise Support is not actively in use, consider whether this role can be restricted or removed.

Risk

  • The role has access to SSM, S3, EC2, CloudWatch Logs, and Directory Services, all without resource-level restrictions.
  • If this role is assumed by an unauthorized party or over-assigned to EC2 instances, it could be used to access SSM Parameter Store secrets, read/write S3 buckets, or interact with managed instances beyond the intended scope.
  • The combination of ec2-instance-connect and broad SSM permissions significantly expands the potential for lateral movement across the EC2 fleet.

Recommendation

  • Scope S3 permissions to specific bucket ARNs required for SSM operations (e.g., SSM session logs bucket).
  • Scope SSM permissions to specific instance ARNs or use resource tags as conditions.
  • Replace the broad Resource: * with specific ARN patterns wherever possible.
  • Implement IAM Condition Keys (e.g., aws:ResourceTag) to restrict access to only authorized instances.
  • Regularly audit which EC2 instances have this role attached via AWS Config or IAM Access Analyzer.

Conclusion

All seven findings in this assessment share a common root cause: IAM policies granting excessive permissions scoped to all resources (*). While some of the affected roles are AWS-managed service-linked roles with inherently broad permissions by design, each should be validated to ensure the policy scope is no greater than what AWS documents as necessary. The user-managed identities (ai_scanner_user, terraform-bootstrap, and EC2InstanceConnectRole) represent the most actionable and highest-priority remediation targets, as their permissions can be directly and immediately reduced without impacting core AWS service functionality.

Immediate action is recommended on all HIGH severity findings.
